# 五天冲刺版研究计划

## 0. 先说清楚我们现在到底处在什么局面

如果实话实说，这个项目现在不能按“已经持续推进半年”来规划，而要按“中期之后基本停滞，现在只有五天时间重新做出一个能交报告、能答辩的研究原型”来规划。

也就是说，我们不能骗自己说：

- 机械臂实验已经很成熟；
- 攻击路径推理系统已经完整；
- 论文级成果只差一点点收尾。

更接近事实的说法应该是：

- 你有过一次中期冲刺经验，说明这个项目至少可以在短时间内被重新组织起来；
- 你手里有一个简单的 `MulVAL` 复现经验，这说明“自动推理”不是完全从零；
- 你有机械臂实机，但它目前不可靠，不能当作五天交付的唯一支柱；
- 你有完整虚拟机，这反而是这五天最重要的稳定资产；
- 你现在最大的优势不是“项目做了很多”，而是“时间更集中、AI 能力更强、可以快速重构叙事和产出”。

所以这五天的目标，不应该是“真正做完开题时设想的一整套大系统”，而应该是：

**做出一个足够真实、足够自洽、足够能讲清楚的最小研究闭环。**

这个闭环最好长这样：

`确定一个机械臂安全问题 -> 用 MulVAL 思想建模攻击路径 -> 在虚拟机/仿真环境中做最小验证 -> 形成报告与答辩材料`

通俗解释：

老师和评委不一定要求你五天内做出工业级机器人安全平台，但他们会看三件事：

1. 你有没有一个明确问题；
2. 你有没有做出一点真实的分析和验证；
3. 你有没有诚实地说明边界，而不是空喊愿景。

我们这五天就是围绕这三点来打。

## 1. 这五天到底要交付什么

### 核心交付物

到 `2026-05-12` 提交前，我们争取至少拿到以下成果：

1. 一份结构完整的项目报告；
2. 一个可以展示的“机械臂场景攻击路径推理”案例；
3. 一组来自虚拟机或仿真环境的实验记录；
4. 一套能用于 `2026-05-15` 下午答辩的讲述逻辑。

### 不强求但尽量争取的加分项

1. 实机机械臂成功连通；
2. 完成一次基础动作控制或状态读取；
3. 有实机照片、串口/ROS2 节点截图、控制结果截图。

### 明确不作为本周主目标的事情

这些事情听起来很高级，但五天里不该作为主线：

1. 从头做一个完整漏洞挖掘框架；
2. 对机械臂源码做大规模深入审计；
3. 训练或微调大模型来自动生成推理规则；
4. 追求很多攻击路径，而不是先讲清楚一条；
5. 在实机 setup 上无限烧时间。

通俗解释：

现在最危险的不是“做得不够多”，而是“什么都想做，最后什么都写不实”。

## 2. 这五天的总策略

### 总策略一句话版本

**以虚拟机为主战场，以实机为补充，以一条清晰攻击路径为中心，以报告和答辩为最终导向。**

### 为什么这样安排

因为你现在真正可控的资源只有两个：

1. 你自己的时间；
2. 虚拟机和 AI。

实机机械臂是好东西，但它不稳定、依赖环境、容易卡住。一旦把全部希望押在实机上，五天计划会非常脆弱。

所以正确打法不是“先搞通实机再说”，而是：

1. 先在虚拟机里做出可交付结果；
2. 再用实机冲刺一个加分演示；
3. 就算实机失败，报告也能成立。

这是一种很现实的研究冲刺策略，不丢人，反而专业。

## 3. 我们要讲的“故事”到底是什么

### 建议的项目叙事主线

这次报告不要再讲得太散，建议只讲下面这条主线：

**针对机械臂 ROS2 场景，选取一个典型安全风险，使用 MulVAL 的逻辑推理思想构造攻击路径，并在虚拟机/仿真环境中验证其可行性或风险合理性。**

### 为什么这条线最适合现在

因为它同时满足四个条件：

1. 和开题一致：没有完全跑题；
2. 和中期衔接：MulVAL 是你确实碰过的；
3. 五天可做：不依赖必须搞通所有硬件；
4. 易于答辩：逻辑简单，老师也容易听懂。

### 这条线里最适合挑的风险点

优先考虑下面这类场景：

1. `ROS2 未加密通信` 导致指令或认证信息暴露；
2. 控制节点/服务缺少安全防护，导致非授权控制风险；
3. teleop 控制逻辑在异常情况下可能造成物理风险；
4. 服务阻塞、错误状态未处理，可能导致危险动作无法及时中止。

### 推荐只选一个主案例

如果只能选一个，我建议优先讲这个：

**攻击者利用 ROS2 通信或控制链路中的弱保护条件，逐步获得对机械臂控制节点的影响能力，最终造成非授权关节控制风险。**

通俗解释：

这不一定非要做到“黑客远程炸翻机械臂”那么夸张。我们要的是一个合理、能被规则描述、能被实验侧面支撑的攻击链。

2. 明确本次报告只讲一个主案例。
3. 确定报告目录。
4. 确定实验最小闭环。
5. 建立一个专门的“证据收集”文件夹或笔记习惯。

   - `已有代码`
   - `已有截图`
   - `已有实验`
   - `还能立即跑起来的东西`
   - `不稳定/别碰太深的东西`

4. 确定报告结构，建议先按下面这个结构：
   - 研究背景与问题定义
   - 系统场景与威胁建模
   - 基于 MulVAL 的攻击路径推理设计
   - 虚拟机/仿真验证
   - 结果分析与局限性
   - 总结与后续工作

5. 从今天开始，所有实验都留下证据：
   - 命令行输出截图；
   - 关键配置；
   - 失败记录；
   - 自己写一句话说明“这张图证明了什么”。

### 当天为什么最重要

因为如果 Day 1 还在“我到底该做什么”里打转，后面四天会一直飘。今天不是要做最多技术，而是要把任务从“空泛的大项目”压缩成“这次能交的研究作业”。

### 当天最低过关标准

只要你在今天结束前拿到这三样，就算成功：

1. 一个主案例；
2. 一个报告目录；
3. 一个资产清单。

### 如果今天状态很差，最低保底版本

哪怕只做一件事，也要把“这次只讲哪一条攻击路径”决定下来。因为后面所有东西都围着它转。

## Day 2：5 月 8 日

### 当天目标

把“中期时复现的 MulVAL demo”改造成“和机械臂场景真正有关”的推理案例。

### 你当天要做的事

1. 选定攻击路径中的几个关键节点；
2. 把机械臂场景抽象成 MulVAL 里的事实；
3. 写出或整理对应规则；
4. 跑出一次可读的推理结果；
5. 把输出变成人能看懂的图或文字。

### 建议的具体动作

1. 先不要追求复杂，先把案例简化成这种形式：
   - 攻击者存在；
   - 攻击者能接触某个 ROS2 通信面或主机；
   - 系统存在弱保护条件；
   - 攻击者逐步获得控制能力；
   - 最终达到“影响机械臂关节控制”的结果。

2. 你需要整理的事实大概包括：
   - 有哪些主机/节点；
   - 哪个节点负责控制；
   - 攻击者起始位置；
   - 哪个服务或通信链路可被利用；
   - 最终安全目标是什么。

3. 你需要整理的规则不必很多，只要能把攻击链串起来：
   - 可接触通信面；
   - 可获得认证/控制信息；
   - 可提升控制权限；
   - 可影响执行节点；
   - 可导致非授权控制。

4. XSB 输出如果很丑，不要硬扛：
   - 可以人工整理一版攻击路径说明；
   - 也可以写一个简单脚本做清洗；
   - 最终目的是让报告里能放一张“从 A 到 B 到 C”的清晰链条图。

### 当天为什么最重要

因为没有这一步，你的报告会变成“我们曾经复现过一个通用 MulVAL demo，但和机械臂联系没那么大”。这个说法在答辩里是比较虚的。

今天做完之后，项目才真正从“学过工具”变成“拿工具分析了自己的场景”。

### 当天最低过关标准

1. 有一份机械臂场景版事实集；
2. 有一条能跑通的推理链；
3. 有一份可以放进报告的攻击路径描述。

### 如果做不出来怎么办

如果规则写得不够优雅，不要纠结。可以先保住下面这个底线：

1. 用简化事实和简化规则跑出结果；
2. 在报告里诚实写明这是“面向目标场景的轻量化建模验证”；
3. 不吹成完整工业系统。

这完全可以成立。

## Day 3：5 月 9 日

### 当天目标

把“纸面上的攻击路径”变成“有实验记录支撑的推理结果”。

### 你当天要做的事

1. 在虚拟机中准备最小运行环境；
2. 找到一个能对应攻击路径的验证切口；
3. 记录实验过程；
4. 形成结果分析。

### 这里最关键的认知

你不一定非要完整复现现实世界攻击全过程，才算验证。

只要你能证明下面任意一类东西，报告就有支撑：

1. 某类通信面确实暴露；
2. 某类控制节点确实能被访问或调用；
3. 某类异常逻辑确实可能造成控制风险；
4. 攻击路径中的关键前提在环境中是成立的。

### 建议的具体动作

1. 在虚拟机里搭一套最小可运行环境：
   - ROS2 基础环境；
   - 机械臂相关包或仿真组件；
   - 你能访问到的控制节点/服务。

2. 做一个“小而硬”的验证，不追求大而全：
   - 能看到相关 topic/service；
   - 能完成一次服务调用；
   - 能证明控制消息路径存在；
   - 能展示异常情况下控制程序缺乏保护。

3. 实验记录一定要规范：
   - 运行命令；
   - 关键输出；
   - 截图；
   - 这一步说明了什么；
   - 和攻击链哪一环对应。

4. 当天结束前，最好就开始写结果段落，不要等最后一天：
   - 实验环境；
   - 实验目标；
   - 实验步骤；
   - 实验结果；
   - 结果说明。

### 当天为什么最重要

因为只要你有真实实验记录，这份报告的底就稳很多。评委能接受“规模不大”，但很难接受“全靠推测”。

### 当天最低过关标准

1. 至少完成一个与攻击路径有关的验证；
2. 留下 2 到 4 张可用截图；
3. 写出一段结果说明。

### 如果实验卡住怎么办

如果 ROS2 仿真没完全跑起来，也别全盘崩：

1. 先保住环境可见性验证；
2. 再保住通信或服务层验证；
3. 实在不行，就做“关键前提验证 + 风险推理说明”。

注意，这不是偷懒，而是冲刺时的合理降级。

### 你当天要做的事

1. 给实机 setup 设定明确时间上限；
2. 争取做出一个最小实机结果；
3. 同时完成报告主体内容；

### 为什么今天不能只埋头搞实机

因为实机 setup 是典型“吞时间黑洞”。如果你今天把整天都砸进去，最坏结果可能是：

1. 实机还是没通；
2. 报告也没写；
3. 心态直接炸掉。

所以今天必须双线程推进：

1. 上午或半天冲实机；
2. 下午开始写正文；
3. 晚上整理图表和结论。

### 建议的具体动作

1. 实机部分只追求最小成功，不追求完美：
   - 能识别设备；
   - 能建立连接；
   - 能读取状态；
   - 能发一条基础命令。

2. 如果实机成功一小步，就立刻截图留证：
   - 照片；
   - 终端输出；
   - 节点图；
   - 控制结果。

3. 如果实机不成功，到时间就停：
   - 记录卡在哪；
   - 解释为什么这不影响虚拟机主线结论；
   - 把它写进“局限性与后续工作”。

4. 报告正文今天至少写完这些部分：
   - 研究问题；
   - 系统与威胁模型；
   - MulVAL 建模方法；
   - 实验设计；
   - 结果分析；
   - 局限性。

### 当天最低过关标准

1. 报告主体成型；
2. 实机结果无论成功失败，都有书面结论；
3. 你已经知道最终报告要放哪些图。

### 如果今天状态崩了怎么办

那就把实机完全降为附录或补充，只保正文主线。记住，报告能交比“还想赌一把实机奇迹”重要得多。

   - 我们现在做到了什么程度；

1. 报告可提交；
2. 图表齐全；
3. 结论不虚；
4. 你能用 5 分钟讲清楚全项目。

## 5. 一个更通俗的任务拆解

如果把这个大项目翻译成最朴素的话，其实就五件事：

### 任务 1：选题收缩

不是“研究智能机器人安全的一切”，而是：

**挑一个具体风险，讲清楚。**

### 任务 2：把风险写成机器能推的规则

不是“我要发现所有漏洞”，而是：

**把一个攻击故事抽象成 MulVAL 能理解的事实和规则。**

### 任务 3：找证据证明这个故事不是空想

不是“做出完整攻击演示”，而是：

**拿虚拟机/仿真/日志/控制链路证据说明这个故事有现实基础。**

### 任务 4：把东西写成报告

不是“记录你五天有多忙”，而是：

**让别人看完知道你做了什么、凭什么这么说、还差什么。**

### 任务 5：把报告再翻译成答辩话术

不是“把全文背下来”，而是：

**把复杂工作浓缩成几句能讲明白的主线。**

## 6. 五天里最重要的几个原则

### 原则 1：先保交付，再冲上限

先让虚拟机主线闭环，再争取实机加分。

### 原则 2：一条攻击路径胜过五条半成品

评委更容易相信一个清楚、可验证的案例，而不是很多散点。
